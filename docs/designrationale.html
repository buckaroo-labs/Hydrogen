This document will explain the design choices meant to support the requirements as defined in the Requirements document.

These are some of the project's guiding ideas:

  • Every additional piece of software that a developer has to download or configure is a barrier to productivity.
  • Every service that a developer has independently register for is a barrier to productivity.
  • Have default stylesheet in Hydrogen directory which can be overridden by separate stylesheet file at web root
  • Set nav- menu-bar sizes in global settings; have template CSS driven by PHP to set margins, widths, and heights accordingly
  • Use SQLite as the default RDBMS

Bot prevention

The user login and registration forms present an attack surface for malicious activity. One common way of preventing usage of such forms by automated attacks is to implement CAPTCHA. The two greatest drawbacks of such a system are that even the most simple CAPTCHA challenges can be a burden on the user, and that their implementation is always a burden on the developer, often requiring registration with some provider.

For these reasons, this project is trying a different approach. All forms will include a hidden input containing an integer value. This value will be updated by a JavaScript function, and only form data having the updated value will be processed. Bots typically will not run JavaScript and are merely looking for easy targets or low-hanging fruit.

This strategy will not prevent abuse abetted by a human analyst, so that risk ought to be assessed for any given application being built on this framework.

User registration

Sending a password reset or email confirmation link to a user should be easy, but it isn't. The mail server that you pay for, that you supply your own account credentials to, and that obliges you every step of the way through the SMTP message-acceptance protocol may decide that your message looks too much like spam, and consequently discard it as soon as it is done talking to you.

It's an affront to good sense, but to get along in the current world of shared hosting and overwhelming amounts of malicious email, your best option is not to send anything that even looks like a link so that you don't waste hours of your time wondering why messages aren't being delivered.

For this reason, users registering an email address or requesting a password reset will be sent a single-use alphanumeric code for that purpose.
