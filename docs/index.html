Hydrogen is a Framework for development and documentation of responsive web applications. It is based on a w3schools example, using w3.css, and has been in (occasional) development since 2014.

Hydrogen is designed for small projects that run on inexpensive hosting, e.g. a LAMP stack, and runs on vanilla PHP. Data storage options include SQLite, MySQL, and (sort of) Oracle. The guiding principle of this project is to reduce barriers to productivity so that developers can focus on creating their particular application rather than repeating steps that are common to many applications.

This document is built on HDoc, a tool built specifically for the documentation of the Hydrogen project, but easily repurposed elsewhere. See the HDoc meta-documentation for more about this document's layout and styling.

This document is one of multiple places that the project is documented in. For issues and planned features, the github repository will be a better source. For the most low-level implementation details, see the comments in the code itself.

Hydrogen is designed to support the following high-level functional requirements:

  • Consistent look for multi-page applications
  • Dynamic menu and navigation bars
  • Simple identity and access management with role-based access control
  • Persistent login through the use of token cookies (JWT)
  • Simplified creation of HTML tables from SQL data sources
  • Pagination of table data
  • Logging of debugging output

Hydrogen is designed to support the following high-level non-functional requirements:

  • Usability on both desktop and mobile browsers
  • Ease of setup
  • Security
  • Extensibility via APIs
  • Customization via CSS, JS, PHP
  • Clearly-structured and -commented code
  • Up-to-date library compatibility

User registration

  • Prevent bot traffic
  • A system-generated username is assigned upon registration of email address.
  • To complete registration, users must supply a first and last name.

Account self-service

  • Users are able to request a reset of their own password.
  • Users may choose their own username, with reasonable limitations.
  • Users may set a profile image for their account.
  • Users may change their first and last names and/or email address.

Password Authentication

  • Support password age, reuse, length and complexity requirements
  • Allow users to log in using either their username or email address.
