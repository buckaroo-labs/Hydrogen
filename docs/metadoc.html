HDoc is short for Hydrogen Documentation, a template for creating technical documentation with HTML, CSS and Javascript.

HDoc was originally designed to support the documentation of the Hydrogen framework for web applications, and the contraction HDoc serves to distinguish the documentation for the application development framework from the meta-documentation.

In-progress features listed below are highlighted like this.

Before getting into how HDoc works, it would be best to document what it is supposed to do. Going from generalities to specifics, HDoc aims to:

  • Provide a useful template
  • Reduce redundancies in the editing process
  • Automatically generate a table of contents with hyperlinks to document headings
  • Manage the numbering of endnotes and citations
  • Provide a responsive CSS framework for illustrations that float among text
  • Highlight the completion status of document sections and of the associated work

The general idea is to avoid proprietary document formats and editing software. MS Word is a fine product, but I prefer the control that comes with using HTML. My chief disappointment with Word is that even though it offers an HTML option, it is so bloated as to make the document source unreadable.

The template provided by HDoc will readily support customization.

To improve both the user experience and ease of development, the Bootstrap library will be included in the document template.

For ease of navigation, the document will have a top-of-page link that is always visible in the browser.

I'm not a fan of deep nesting of outlined elements on a single page. If you need to go more than two levels deep, create links from your index page to more specific documents. In a future release, HDoc will provide the user with navigation buttons to go to the index page or back up the heirarchy if you define one or more A elements (a link or links) on these specific documents with the ID attribute of HDocHome or HDocUp.

The following are the hows relating to the whats defined in the above requirements.

A default visual style of the document will be defined in the file hdoc.css.

To reduce redundancy, an H1 heading will be created from the document's title tag.

H2 headings will be created automatically from the id attributes of top-level document elements. Top-level elements are defined by using the DIV tag with a class attribute of contentsL2 (The top level is level two, because the Title element is shown as an H1; too bad that HTML heading levels aren't numbered starting with zero).

These H2 headings will be automatically added to a table-of-contents element which will be added to the document's beginning.

Indexed content that is lower in the document heirarchy will be nested in the top-level element as another DIV element with the class contentsL3. H3 headings will be generated for these in the same manner as H2 headings, and will also be added to the table of contents.

Heading generation will be accomplished via Javascript in the file hdoc.js.

The file style.css, which is provided as a blank, will be linked so that the document's default style can be overridden without alterations to the file hdoc.css.

Automatically generated page content will have CSS classes (beginning with the letters HDoc ) so that this content can be selected and customized using the styles.css file.

The file hdoc.js will add a top-of-page link to the bottom of the browser window.

All <q>, <blockquote> and <span> elements having a cite attribute will have an <a> element appended after them showing a superscripted, sequential number and hyperlinked to the URL specified in the cite attribute. This number and its associated URL will also be appended to an End Notes section of the document, which will be produced automatically.

All <end-note> elements will have an <a> element appended after them showing a superscripted, sequential number and hyperlinked to a same-numbered entry in the End Notes section showing the value of the <end-note> title attribute. For example: Cool Prius

<inline-note> elements will be rendered as red, bold, double asterisks. When clicked, they will display the value of the <inline-note> element's title attribute.

To recap, the cite attribute is used with URL citations, and other citations or notes use the title attribute.

In-progress items identified with the inprogress class are highlighted in yellow.

Incomplete document sections identified with the incomplete class are highlighted in black.

CSS classes will be provided for setting screen width thresholds for illustration elements' float and width attributes and the grid-template-columns attribute of their parent containers. See Usage below.

Code for the top-of-page button is adapted from https://www.w3schools.com/howto/tryit.asp?filename=tryhow_js_scroll_to_top .

To create your own content using HDoc:

  1. Copy this file.
  2. Modify the page title in the <head>.
  3. Delete the content between the <body> tags.
  4. For each top-level section of your document, add a DIV element to the body with a class attribute of contentsL2 and an id attribute containing the section's intended title. Replace any spaces in the ID/title with an underscore character.
  5. For each lower-level section of your document, add a DIV element within the relevant top-level element with a class attribute of contentsL3 and an id attribute containing the section's intended title. Replace any spaces in the ID/title with an underscore character.
  6. Customize the document's appearance by adding your own CSS in the style.css file.

You may have two different document sections that you want to give the same subheading, but you want them to have unique IDs for bookmarking in the table of contents. Set the element IDs to the text you want, and then give each ID a unique suffix starting with two underscore characters. This suffix will not be displayed in the heading.

If your document has figures that float to the left or right of text, it is helpful to designate a minumum screen width to allow floating, below which the figure will occupy most or all of the width of its parent container. To this end, HDoc provides CSS classes specifying this minumum screen width and media queries at the matching widths. Use the figure.css stylesheet and the classes nofloat_500, nofloat_600, nofloat_750, nofloat_800, nofloat_900, and nofloat_1100. The woodcut figure in this section is set to float at a minimum of 900 pixels; try it out if your screen is that large.

It is also helpful to specify screen width thresholds at which figures occupy smaller width percentages of wider screens. See the figure.css stylesheet for a complete list of the classes available for this purpose. As an example, applying the classes float_600_50, float_900_33, float_1200_15 to your element will result in that element occupying 50% of the width of its parent at a minimum of 600px, 33% at 900px, and 15% at 1200px; the exception being when a nofloat class applies at that width, in which case it will take precedence. Because the float classes all require a minimum width, it is best to use a nofloat class that specifies a larger width than the smallest width specified in your element's float classes.

Lastly, it is helpful to make the number of columns in a grid layout depend on screen width. HDoc helps you to easily specify the minimum screen widths for two, three or four columns. See the figure.css stylesheet for a complete list of the classes available for this purpose. For example, adding the g2-800 and g3-1200 classes to your container element having a default of one column will set it to two columns at a minimum width of 800px and to three columns at a minimum width of 1200px.
